How It Works — DAA Algorithms Design & Analysis of Algorithms — core concepts implemented

1. Rule Parsing — O(n)

Each rule is tokenized and normalized into a structured object: action, protocol, source CIDR, source port range, destination CIDR, destination port range. CIDR notation is converted to numeric ranges for fast comparison.

function parseRule(line) {
  const [action, proto, src, dst] = line.split(' ');
  return { action, proto,
    srcNet: cidrToRange(src),
    dstNet: cidrToRange(dst) };
} // O(1) per rule → O(n) total

2. Conflict Detection — O(n²)

Two rules conflict if their IP ranges and port ranges overlap but they have different actions (ALLOW vs DENY). For each pair we check IP range intersection and port range intersection using interval overlap logic.

function overlaps(a, b) {
  // IP range overlap
  const ipOvlp = a.ipStart <= b.ipEnd
    && b.ipStart <= a.ipEnd;
  const ptOvlp = a.portStart <= b.portEnd
    && b.portStart <= a.portEnd;
  return ipOvlp && ptOvlp;
}

3. Redundancy Detection — O(n²)

Rule B is redundant if Rule A has the same action and its IP+port range fully contains B's range (A is a superset). Rule A subsumes Rule B — B adds no new behavior to the firewall.

function isSubsumed(broad, narrow) {
  return broad.action === narrow.action
   && broad.ipStart <= narrow.ipStart
   && broad.ipEnd >= narrow.ipEnd
   && broad.portStart <= narrow.portStart
   && broad.portEnd >= narrow.portEnd;
}

4. Shadowing Detection — DFS on DAG

Rule B is shadowed if Rule A appears first AND A matches every packet B would match. We build a dependency DAG and run Depth-First Search (DFS) to find all rules that are unreachable — they can never be evaluated.

function buildDAG(rules) {
  const graph = new Map();
  for (let i=0; i<rules.length; i++)
    for (let j=i+1; j<rules.length; j++)
      if (shadows(rules[i],rules[j]))
        graph.get(i).push(j);
  return graph;
} // DFS: O(V+E)

5. Topological Sort — O(V+E)

Before reordering, we topologically sort the rule dependency graph to determine a valid ordering. More-specific rules (smaller CIDR prefix, specific port) receive higher priority and are moved earlier in the chain.

function topoSort(graph, n) {
  const visited = new Set();
  const result = [];
  function dfs(node) {
    if (visited.has(node)) return;
    visited.add(node);
    graph[node].forEach(dfs);
    result.unshift(node);
  }
  return result;
}

6. Greedy Optimization — O(n log n)

Rules are scored by specificity (prefix length + port specificity). Using a greedy approach, we sort rules by descending specificity score. More specific rules are placed first — ensuring the most targeted rules are evaluated before broad catch-all rules.

function specificity(rule) {
  const ipScore = rule.prefixLen / 32;
  const ptScore = rule.port !== 'ANY'
    ? 1 : 0;
  return ipScore + ptScore;
}
rules.sort((a,b) =>
  specificity(b) - specificity(a));

7. Hashing for Duplicate Detection — O(n)

Exact duplicate rules are detected in O(n) time by hashing each rule's normalized string representation into a hash map. Rules with the same hash (and same content) are flagged as exact duplicates and removed in one pass.

function removeDuplicates(rules) {
  const seen = new Map();
  return rules.filter(r => {
    const key = hashRule(r);
    if (seen.has(key)) return false;
    seen.set(key, true);
    return true;
  }); // O(n) total
}

8. CIDR Interval Tree — O(n log n)

IP ranges expressed in CIDR notation (e.g. 192.168.0.0/16) are converted to numeric intervals [start, end]. An Interval Tree allows finding all overlapping ranges in O(log n) per query — far better than the O(n) linear scan.

function cidrToRange(cidr) {
  const [ip, len] = cidr.split('/');
  const base = ipToInt(ip);
  const mask = ~((1 << (32-len))-1);
  return {
    start: base & mask,
    end: (base & mask) | ~mask
  };
}
